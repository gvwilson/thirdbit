Interactive Dashboards

Learning Goals

  • Explain how a dashboard differs from a static chart file
  • Build a FastHTML app that serves an Altair chart from a running web server
  • Use HTMX attributes to update one part of a page without reloading it
  • Trace the request-response cycle when a dropdown filter changes

Why Static Charts Are Not Enough

  • A PNG or a saved HTML chart answers one specific question with one specific slice of data
    • A colleague who wants to see work hours for Java only, then for Perl only, needs you to regenerate and re-send the chart each time
    • A chart in a paper is a frozen snapshot; a dashboard is a live view
  • A dashboard serves charts from a running web server
    • Collaborators filter and explore the data themselves without writing any code
    • The researcher who built the dashboard controls what data is available and what filters make sense
  • The Prechelt data from Lesson 1 is a good example
    • Seven languages, widely varying work hours, a question every reader wants to ask: "What does this look like for just the language I care about?"
    • Building that filter into a dashboard takes about fifty lines of Python

How FastHTML Works

  • FastHTML is a Python web framework: you write Python functions, it generates HTML
    • A route maps a URL path to a Python function; when a browser visits that path, the function runs and its return value becomes the response
    • HTML elements are Python constructors: Div(P("hello")) produces <div><p>hello</p></div>
    • No template files, no HTML to write by hand, no string formatting
  • Server-side rendering means the HTML is built on the server for each request
    • The browser receives complete HTML and displays it immediately
    • All data access and computation happen in Python on the server, where you already know how to work

HTMX: Interactivity Without JavaScript

  • Normally, a browser interaction (selecting a dropdown, clicking a button) either does nothing or reloads the entire page with a new URL
  • HTMX changes that: add a few attributes to an HTML element and it will send a server request when something happens, then replace one targeted element on the page with the response
    • The whole page does not reload; only the element you name is updated
    • The server returns an HTML fragment, not a full page
  • FastHTML exposes HTMX attributes as Python keyword arguments: hx_get="/chart" becomes hx-get="/chart" in the rendered HTML
  • Three attributes cover the most common pattern:
    • hx_get="/path": send a GET request to this path when the trigger fires
    • hx_target="#element-id": put the response into the element with this CSS id selector
    • hx_trigger="change": fire when this HTML event occurs (change, click, input, load, and others)

A Minimal FastHTML App

  • fast_app() creates the application and routing table and returns two objects: the app itself and a route decorator
  • Decorate a function with @rt("/") to make it handle GET requests to the root URL
  • Titled("Title", content) produces a page with a heading and body content
  • serve() starts the development server on http://localhost:5001; stop it with Ctrl-C
from fasthtml.common import *

app, rt = fast_app()

@rt("/")
def get():
    return Titled("Hello", P("Welcome to my dashboard"))

serve()
  • Run this file and open http://localhost:5001 in a browser to see the result

Adding a Chart

  • Install fh-altair with uv add fh-altair; it bridges FastHTML and Altair
  • Pass hdrs=altair_headers to fast_app() to load the Vega-Lite JavaScript libraries in the page <head> — without this, the chart renders as a blank rectangle
  • altair2fasthtml(chart) converts an Altair chart object to an HTML div that the browser renders as an interactive Vega-Lite chart
from fasthtml.common import *
from fh_altair import altair_headers, altair2fasthtml
import altair as alt
import polars as pl

app, rt = fast_app(hdrs=altair_headers)

@rt("/")
def get():
    df = pl.read_csv("data/jccpprtTR.csv")
    chart = (
        alt.Chart(df)
        .mark_boxplot()
        .encode(
            x=alt.X("lang:N", title="Language"),
            y=alt.Y("whours:Q", title="Work hours"),
        )
        .properties(width=560, height=320)
    )
    return Titled("Work Hours", altair2fasthtml(chart))

serve()
  • Altair 6 accepts Polars DataFrames directly; no conversion to Pandas is needed
  • The chart is interactive out of the box: scroll to zoom, drag to pan, hover for tooltips

Making It Interactive

  • A Select element with HTMX attributes sends the selected value to the server when the user changes the dropdown
  • The server function receives the value as a plain Python parameter
  • The server filters the data and returns a new chart fragment, which HTMX swaps into the #chart div without touching the rest of the page
@rt("/")
def get():
    df = pl.read_csv("data/jccpprtTR.csv")
    options = [Option("All languages", value="All")] + [
        Option(lang, value=lang) for lang in LANGUAGES
    ]
    return Titled(
        "Development Time Explorer",
        Div(
            Label(
                "Filter by language: ",
                Select(
                    *options,
                    name="language",
                    hx_get="/chart",
                    hx_target="#chart",
                    hx_trigger="change",
                ),
            ),
            Div(altair2fasthtml(build_chart(df)), id="chart"),
        ),
    )

@rt("/chart")
def get(language: str = "All"):
    df = pl.read_csv("data/jccpprtTR.csv")
    if language != "All":
        df = df.filter(pl.col("lang") == language)
    return altair2fasthtml(build_chart(df))
  • When HTMX sends GET /chart?language=Java, FastHTML converts the query parameter to the language argument automatically
  • The default value language: str = "All" prevents an error when the page first loads and /chart has not yet been called with any argument
  • The id="chart" on the outer Div is the target that hx_target="#chart" refers to; the # prefix is the CSS selector for an element with that id

"""Interactive dashboard for exploring Prechelt development-time data."""

from fasthtml.common import Div, Label, Option, Select, Titled, fast_app, serve
from fh_altair import altair_headers, altair2fasthtml
import altair as alt
import polars as pl

LANGUAGES = ["Ada", "C", "C++", "Java", "Perl", "Rexx", "Tcl"]

DATA_PATH = "data/jccpprtTR.csv"


def build_chart(df):
    """Return a box plot of work hours per language."""
    return (
        alt.Chart(df)
        .mark_boxplot()
        .encode(
            x=alt.X("lang:N", title="Language"),
            y=alt.Y("whours:Q", title="Work hours"),
        )
        .properties(width=560, height=320, title="Development time by language")
    )


app, rt = fast_app(hdrs=altair_headers)


@rt("/")
def index():
    df = pl.read_csv(DATA_PATH)
    options = [Option("All languages", value="All")] + [
        Option(lang, value=lang) for lang in LANGUAGES
    ]
    return Titled(
        "Development Time Explorer",
        Div(
            Label(
                "Filter by language: ",
                Select(
                    *options,
                    name="language",
                    hx_get="/chart",
                    hx_target="#chart",
                    hx_trigger="change",
                ),
            ),
            Div(altair2fasthtml(build_chart(df)), id="chart"),
        ),
    )


@rt("/chart")
def chart(language: str = "All"):
    df = pl.read_csv(DATA_PATH)
    if language != "All":
        df = df.filter(pl.col("lang") == language)
    return altair2fasthtml(build_chart(df))

Check Understanding

Trace what happens in the browser and on the server when a user changes the dropdown from "All languages" to "Java".

The browser fires a change event on the Select element. HTMX intercepts it and sends a GET request to /chart?language=Java. The server receives the request, FastHTML routes it to the second get function, and the language parameter is set to "Java". The function loads the CSV, filters it to Java rows only, builds a box plot of those rows, converts it to an HTML div with altair2fasthtml, and returns it. HTMX receives the HTML fragment and replaces the contents of the element with id="chart" with the new chart. The rest of the page — the heading, the dropdown — is unchanged.

The code below does not update the chart when the dropdown changes. What is wrong and how do you fix it? 
Select(
    *options,
    name="language",
    hx_get="/chart",
    hx_target="chart",
    hx_trigger="change",
)

hx_target="chart" looks for an element whose tag name is chart, which does not exist in HTML. HTMX targets use CSS selectors: a bare word is a tag name, and an id must be prefixed with #. The fix is hx_target="#chart", which correctly targets the Div whose id is "chart". Without the #, HTMX finds nothing to update and silently does nothing.

Why does the /chart route need a default value for the language parameter?

When the page first loads, only the root route / is called. The Div with id="chart" is populated directly by calling build_chart(df) inside the / handler. The /chart route is only triggered by HTMX when the dropdown changes, and HTMX always sends the current dropdown value as a query parameter. However, FastHTML would raise an error if a request arrived at /chart without a language parameter and the function had no default. Providing language: str = "All" as a default means the route is safe to call even if the parameter is missing — for example, when testing the route directly in the browser.

What goes wrong if you forget to pass hdrs=altair_headers to fast_app()?

altair_headers loads the Vega, Vega-Lite, and vegaEmbed JavaScript libraries in the page <head>. The altair2fasthtml function produces an HTML div with a short script that calls vegaEmbed to render the chart specification. If those libraries are not loaded, vegaEmbed is undefined and the script throws a JavaScript error. The div that was supposed to contain the chart is blank. The rest of the page loads normally, which makes the problem easy to miss: everything appears to work except that the chart area is empty.

Exercises

Add a Work-Hours Range Filter (30 min)

Add two number inputs below the language dropdown: one for a minimum work-hours cutoff and one for a maximum. When either input changes, the chart should update to show only developers whose whours value falls within the range. Use a Polars filter combining both conditions. Verify that setting the range to 0–100 shows all developers and that setting it to 0–10 shows only those who finished quickly.

Show a Summary Card (20 min)

Add a second HTMX target below the chart: a Div with id="summary" that shows the count and median work hours for the currently selected data. Update both the chart and the summary card on each dropdown change. One approach is to have the /chart route return both the chart fragment and the summary card fragment as siblings inside a wrapper Div.

Add a Scatter Plot Tab (30 min)

Add a second route /scatter that returns a scatter plot of whours (x axis) versus stmtL (statement lines, y axis) for the filtered language. Add a pair of buttons — "Box plot" and "Scatter plot" — that each trigger a request to the appropriate route and replace the #chart div. Use hx_trigger="click" on each button.

Show the Five Slowest Developers (20 min)

Below the chart, add a table showing the five developers with the highest work hours in the current selection. Use Polars to sort the filtered data descending by whours and take the first five rows. Build the table using FastHTML's Table, Tr, Th, and Td constructors. Update the table alongside the chart whenever the filter changes.

Share with a Classmate (15 min)

By default, serve() binds to localhost, which means only your own machine can reach it. Change the call to serve(host="0.0.0.0") and find your machine's local IP address. Ask a classmate on the same network to open http://your-ip:5001 in their browser. Verify that their dropdown selections update the chart they see and do not affect the chart you see. Write one sentence explaining why two users can interact with the dashboard independently even though only one Python process is running.

Handle an Empty Filter Result (20 min)

Currently, if a language filter produces zero rows (for example, if someone edits the URL directly), build_chart passes an empty DataFrame to Altair, which may produce an error or a blank chart. Add a check: if df is empty after filtering, return a P element saying "No data for this selection" instead of a chart. Verify by temporarily adding an impossible filter condition (pl.col("whours") < 0) and confirming the message appears.
