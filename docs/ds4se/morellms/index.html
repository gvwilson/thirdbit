More About LLMs

Model Context Protocol

  • Model Context Protocol (MCP) is an open standard for connecting LLM applications to external tools and data sources
    • An MCP server exposes a set of tools, each with a name, description, and schema for its inputs; the LLM sees the descriptions in its context and can choose to call one
    • The client executes the actual call and returns the result; the server does not run the LLM
  • Common MCP servers: filesystem access, SQLite databases, web search, and GitHub
  • The SQLite MCP server is useful for data analysis: ask a question in natural language and the model generates and runs the query automatically
    • Always run the resulting SQL yourself to verify the answer
    • The LLM constructs the SQL; you verify the result; neither step replaces the other
  • MCP decouples the LLM from the tool: the same server works with any MCP-compatible client
  • The server runs as a local process; it does not send your data to a third-party service unless the tool itself makes an outbound request
{
  "mcpServers": {
    "research-db": {
      "command": "uvx",
      "args": ["mcp-server-sqlite", "--db-path", "data/research.db"]
    }
  }
}

Agents

  • A single prompt produces one response; an agent runs a loop: observe, plan, act, and observe again
  • Agents use tool calls to gather information and take action: file reads, code execution, web search, database queries, and shell commands
  • The loop continues until the model decides the task is complete or a maximum step count is reached
  • Agents can take irreversible actions: deleting files, committing code, sending API requests
    • Risk compounds with step count: an error in step two can make every subsequent step wrong
    • Human-in-the-loop checkpoints requiring approval before destructive tool calls limit damage
  • Agents work well for tasks with clear success criteria that can be verified programmatically
    • Running a test suite, querying a database for a specific value, reformatting a set of files
  • Agents work poorly for tasks requiring subjective judgment or an ambiguous environment
  • Always review the full list of actions an agent took before accepting its output

Skills and Extensions

  • A skill is a Markdown file containing a system prompt that specializes the model's behavior for a task
    • A skill can instruct the model to always check documentation before generating code, always output JSON, or always flag when its interpretation implies causation
    • Skills are stored in a configuration directory (e.g., ~/.claude/) and are available across projects
  • Writing a skill automates a prompt pattern you would otherwise repeat by hand every session
$ cat ~/.claude/check-polars.md
# Verify Polars API before generating code

Before generating any Polars code, state which version of Polars you are
targeting and confirm the method signatures against the current documentation.
Do not use Pandas-style index assignment or .groupby() without checking that
these exist in the Polars API.
  • Finding community skills: the Claude Code documentation and GitHub list commonly used ones
  • Installing a skill: copy the .md file to the configuration directory and reference it by name in a prompt or config

Using the Anthropic API

  • Use the API when you need to call an LLM from Python code inside a notebook, script, or pipeline
  • Authentication uses an environment variable, not a hardcoded string in source files
export ANTHROPIC_API_KEY="sk-ant-..."   # add to ~/.zshrc or ~/.bashrc

import anthropic

client = anthropic.Anthropic()   # reads ANTHROPIC_API_KEY automatically
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=512,
    messages=[{"role": "user", "content": "Summarize this dataset in one sentence."}]
)
print(response.content[0].text)
  • The response object contains content (a list of content blocks), usage (token counts), and stop_reason
  • Cost is (input_tokens × input_price) + (output_tokens × output_price); check the provider's pricing page before running large batches
  • Rate limits apply: requests per minute and tokens per minute
    • Exceeded limits raise HTTP 429 errors
    • Use exponential back-off when retrying rate-limited requests: wait 1 second, then 2, then 4, rather than hammering the endpoint
import time

for attempt in range(5):
    try:
        response = client.messages.create(...)
        break
    except anthropic.RateLimitError:
        time.sleep(2 ** attempt)

Exercises

Connect to a Research Database (30 min)

Install the SQLite MCP server and connect it to any SQLite database used in this tutorial. Ask it a natural-language question about the data and record the SQL it generates. Run that SQL yourself and verify the result matches the LLM's answer. Then rephrase the question and record whether the generated SQL changes.

Trace an Agent's Steps (30 min)

Use an LLM agent to find and count all Python files in a project directory. Review every tool call the agent made. Identify one step where the agent made a choice you would have made differently, and write two sentences explaining your alternative and why you prefer it.

Write a Causation-Check Skill (20 min)

Write a two-sentence skill that instructs an LLM to always flag whether its interpretation of a statistical result implies causation or only correlation. Test it on a prompt asking the LLM to interpret a Pearson correlation coefficient. Record whether the skill changed the output compared to the same prompt without the skill.

Call the API from a Script (30 min)

Write a Python script that calls the Anthropic API to summarize a CSV dataset from this tutorial. Print the usage field of the response. Record how many input and output tokens the call consumed and estimate its cost using the published pricing page. Then modify the prompt to ask for a shorter summary and compare token counts.

Handle Rate Limits (20 min)

Modify your API script to add exponential back-off on rate limit errors. Test it by making it retry at least twice. Write one sentence explaining why exponential back-off is preferable to retrying immediately at a fixed interval when many scripts are running concurrently against the same API endpoint.
