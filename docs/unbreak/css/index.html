CSS

CSS controls how HTML elements look on screen. Its rules seem simple at first—select an element, set some properties—but a handful of non-obvious behaviors trip up almost every developer at least once. Open each example directly in a browser to see the problem in action.

Fit a panel inside a container

Open the page. The blue panel should sit entirely within the grey container. Does it, or does it overflow to the right?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Fit a Panel Inside a Container</title>
  <style>
    .container {
      width: 300px;
      background: #ddd;
      padding: 0;
    }
    .panel {
      width: 100%;
      padding: 20px;
      background: steelblue;
      color: white;
    }
  </style>
</head>
<body>
  <h1>Fit a panel inside a container</h1>
  <p>The blue panel should sit entirely within the grey container.
     Does it fit, or does it stick out to the right?</p>
  <div class="container">
    <div class="panel">Panel content</div>
  </div>
</body>
</html>
Show explanation

The bug is box-sizing: content-box, which is the default. When width: 100% is set, the browser assigns 100% of the container's width to the content area alone. Padding is then added on top, making the element wider than the container—300px content + 20px left padding + 20px right padding = 340px total.

Shows: the CSS box model and the difference between content-box and border-box.

To find it: open DevTools, select the panel, and look at the box model diagram in the Computed tab. The total rendered width will exceed the container's width. Add box-sizing: border-box to the .panel rule so that padding is included inside the declared width rather than added to it.

Override a color in a stylesheet

Open the page. Both elements use the .highlight class. Are they both red?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Override a Color in a Stylesheet</title>
  <style>
    nav.highlight {
      color: gray;
    }

    .highlight {
      color: red;
    }
  </style>
</head>
<body>
  <h1>Override a color in a stylesheet</h1>
  <p>The navigation bar below should be red. Is it?</p>
  <nav class="highlight">Site Navigation</nav>
  <p class="highlight">This paragraph uses the same class and should also be red.</p>
</body>
</html>
Show explanation

The bug is a specificity conflict. The selector nav.highlight targets an element type plus a class, giving it specificity (0,1,1). The later rule .highlight is class-only, specificity (0,1,0). CSS resolves conflicts by specificity first and source order second, so the less-specific later rule loses to the more-specific earlier rule even though it appears afterward in the file.

Shows: how the browser calculates specificity and why rule order alone does not determine which style wins.

To find it: in DevTools, click the <nav> element and examine the Styles panel. The .highlight { color: red } rule will be shown with a strikethrough, indicating it was overridden. To fix it, either change .highlight to nav.highlight, .highlight or restructure the stylesheet so both elements share a single rule without the extra type selector.

Add space between sections

Open the page. Section Two has margin-top: 60px and Section One has margin-bottom: 20px. Measure the gap. Is it 80px?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Add Space Between Sections</title>
  <style>
    .section {
      background: #b3d9f7;
      padding: 10px;
      margin-bottom: 20px;
    }
    .section.large-gap {
      margin-top: 60px;
    }
  </style>
</head>
<body>
  <h1>Add space between sections</h1>
  <p>The second section has <code>margin-top: 60px</code> and the first has
     <code>margin-bottom: 20px</code>. How much space actually appears between them?</p>
  <div class="section">Section One</div>
  <div class="section large-gap">Section Two (margin-top: 60px)</div>
  <div class="section">Section Three</div>
</body>
</html>
Show explanation

The bug is margin collapsing. When two block elements are adjacent siblings, their vertical margins do not add up—they collapse into a single margin equal to the larger of the two values. The gap between Section One and Section Two is 60px (the larger margin), not 80px (the sum).

Shows: vertical margin collapsing between adjacent siblings and why the gap is never the sum of both margins.

To find it: in DevTools, hover over the gap between sections. The box model overlay will show only the dominant margin. To get the full 80px gap, convert one of the margins to padding on the element itself, or add display: flow-root to a wrapper to create a new block formatting context that prevents collapsing.

Put a border around an image and caption

Open the page. The blue border should enclose the image placeholder and the caption. Does the border appear to wrap them?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Put a Border Around an Image and Caption</title>
  <style>
    .card {
      border: 3px solid steelblue;
      padding: 8px;
      width: 260px;
    }
    img {
      float: left;
      width: 80px;
      height: 80px;
      background: #aaa;
    }
    .caption {
      float: left;
      margin-left: 12px;
      line-height: 80px;
    }
  </style>
</head>
<body>
  <h1>Put a border around an image and caption</h1>
  <p>The blue border should wrap around both the image placeholder and the caption.
     Does it?</p>
  <div class="card">
    <img alt="placeholder">
    <span class="caption">Photo caption here</span>
  </div>
</body>
</html>
Show explanation

The bug is that floated elements are removed from the normal flow. The parent .card has no in-flow children to give it height, so it collapses to zero height (plus padding). The blue border wraps only the padding, not the floated image or caption.

Shows: how floats interact with parent element height and the classic clearfix problem.

To find it: in DevTools, select .card and observe its height is 16px (just padding). Fix it by adding overflow: auto to .card, which forces the parent to contain its floated children. Alternatively, add a clearing element after the floats, or switch to flexbox or grid.

Show a tooltip above an icon

Open the page. The dark tooltip should appear in front of the red banner. Does it?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Show a Tooltip Above an Icon</title>
  <style>
    .icon {
      display: inline-block;
      width: 40px;
      height: 40px;
      background: steelblue;
      color: white;
      text-align: center;
      line-height: 40px;
      font-size: 20px;
    }
    .tooltip {
      background: #333;
      color: white;
      padding: 4px 8px;
      z-index: 100;
    }
    .banner {
      position: relative;
      z-index: 10;
      background: tomato;
      color: white;
      padding: 8px;
      margin-top: -10px;
    }
  </style>
</head>
<body>
  <h1>Show a tooltip above an icon</h1>
  <p>The dark tooltip should appear in front of the red banner. Does it?</p>
  <div class="icon">?</div>
  <div class="tooltip">Helpful tooltip text</div>
  <div class="banner">I am the banner (position: relative, z-index: 10)</div>
</body>
</html>
Show explanation

The bug is that z-index only works on elements with a position value other than static. The .tooltip element has no position property, so it remains position: static and the z-index: 100 declaration is ignored entirely. The .banner element has position: relative, which enters it into stacking order, so it always paints on top.

Shows: the requirement that z-index needs a non-static position to take effect.

To find it: in DevTools, select .tooltip and look for z-index in the Computed styles. It will show auto (ignored) rather than 100. Add position: relative to .tooltip to activate z-index and let the tooltip stack above the banner.

Add a decorative bullet to each list item

Open the page. Each list item should have a gold star to its left. Do any stars appear?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Add a Decorative Bullet to Each List Item</title>
  <style>
    ul {
      list-style: none;
      padding-left: 0;
    }
    li::before {
      color: gold;
      font-size: 1.2em;
      margin-right: 6px;
    }
  </style>
</head>
<body>
  <h1>Add a decorative bullet to each list item</h1>
  <p>Each item below should have a gold star to its left. Do any stars appear?</p>
  <ul>
    <li>Apples</li>
    <li>Bananas</li>
    <li>Cherries</li>
  </ul>
</body>
</html>
Show explanation

The bug is a missing content property. The ::before pseudo-element exists in the DOM only when content is present—without it the pseudo-element is not generated at all, so no star appears regardless of how color or font-size are set.

Shows: that ::before and ::after require content to exist, and that content: "" (an empty string) is the minimum needed.

To find it: in DevTools, expand the <li> element. No ::before node will appear in the tree. Add content: "\2605" (the Unicode star character) to the li::before rule. Use content: "" as a baseline if you only want to style a generated box with no visible text.

Fill half a page with a colored banner

Open the page. The blue banner should fill half the grey wrapper. Is the wrapper visible at all?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Fill Half a Page with a Colored Banner</title>
  <style>
    .wrapper {
      background: #eee;
    }
    .banner {
      height: 50%;
      background: steelblue;
    }
  </style>
</head>
<body>
  <h1>Fill half a page with a colored banner</h1>
  <p>The blue banner should fill half the grey wrapper. Is the wrapper visible at all?</p>
  <div class="wrapper">
    <div class="banner"></div>
  </div>
  <p>Text below the wrapper.</p>
</body>
</html>
Show explanation

The bug is that percentage heights are resolved relative to the parent's computed height. The .wrapper has no explicit height—its height is auto, which the browser calculates from its in-flow content. The browser cannot resolve 50% of auto, so it treats the child's height as zero. The blue banner has no height and is invisible; the grey wrapper collapses to nothing as well.

Shows: how percentage heights require an explicit height on the ancestor and why auto breaks the chain.

To find it: in DevTools, select .banner and look at its computed height: 0px. To fix it, give .wrapper an explicit height (e.g., height: 300px), use viewport units (height: 50vh) on the banner directly, or switch to a flexbox or grid layout.

Animate a button color on hover

Open the page and hover over the button. Does the background color fade smoothly to red, or does it snap?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Animate a Button Color on Hover</title>
  <style>
    .btn {
      padding: 10px 20px;
      background: steelblue;
      color: white;
      border: none;
      cursor: pointer;
      font-size: 1rem;
      transition: color 0.4s ease;
    }
    .btn:hover {
      background-color: tomato;
    }
  </style>
</head>
<body>
  <h1>Animate a button color on hover</h1>
  <p>Hover over the button. Does the background color fade smoothly to red,
     or does it snap?</p>
  <button class="btn">Hover me</button>
</body>
</html>
Show explanation

The bug is that transition: color 0.4s ease animates the color property (text color), not background-color. The hover rule changes background-color, which has no transition declared, so it jumps instantly. Only a text-color change would animate here, but the hover rule does not change text color, so nothing animates at all.

Shows: how transitions must name the exact CSS property being changed and that color and background-color are distinct.

To find it: in DevTools, look at the .btn rule and the hover rule side by side. The transitioned property (color) does not match the property changed on hover (background-color). Fix it by changing the transition to background-color 0.4s ease or using transition: all 0.4s ease to cover every animated property.

Wrap a grid of cards to a new row

Open the page. Five cards should wrap onto two rows inside the dashed box. Do they wrap, or do they overflow to the right?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Wrap a Grid of Cards to a New Row</title>
  <style>
    .grid {
      display: flex;
      width: 500px;
      border: 2px dashed #999;
    }
    .card {
      min-width: 140px;
      height: 100px;
      background: steelblue;
      color: white;
      margin: 8px;
      display: flex;
      align-items: center;
      justify-content: center;
    }
  </style>
</head>
<body>
  <h1>Wrap a grid of cards to a new row</h1>
  <p>Five cards should wrap onto two rows inside the dashed box.
     Do they wrap, or do they overflow to the right?</p>
  <div class="grid">
    <div class="card">Card 1</div>
    <div class="card">Card 2</div>
    <div class="card">Card 3</div>
    <div class="card">Card 4</div>
    <div class="card">Card 5</div>
  </div>
</body>
</html>
Show explanation

The bug is that flex-wrap defaults to nowrap. All flex children are forced onto a single line regardless of the container's width. When the total minimum size of the children exceeds the container, they shrink as much as they can and then overflow.

Shows: the default flex-wrap: nowrap behavior and why adding flex-wrap: wrap is usually necessary for responsive grid-style layouts.

To find it: open DevTools and select the .grid container. In the Layout panel, the flex wrapping option will show nowrap. Add flex-wrap: wrap to .grid so children that do not fit on one line flow onto subsequent lines.

Add a shadow to a card

Open the page. The white card should have a visible drop shadow. Can you see any shadow?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Add a Shadow to a Card</title>
  <style>
    .container {
      width: 300px;
      background: #f0f0f0;
      padding: 20px;
      overflow: hidden;
    }
    .card {
      background: white;
      padding: 16px;
      box-shadow: 4px 4px 12px rgba(0, 0, 0, 0.4);
    }
  </style>
</head>
<body>
  <h1>Add a shadow to a card</h1>
  <p>The white card should have a visible drop shadow. Can you see any shadow?</p>
  <div class="container">
    <div class="card">Card content goes here.</div>
  </div>
</body>
</html>
Show explanation

The bug is overflow: hidden on the container. box-shadow is painted outside the element's border-box, in the space between the card and the container's edge. Because overflow: hidden clips everything that extends beyond the container's boundary, the shadow—which lives in that clipped region—is invisible.

Shows: that overflow: hidden clips more than just scrollable content; it also clips effects such as box-shadow and outline that paint outside the element's box.

To find it: temporarily set the container to overflow: visible in DevTools. The shadow will reappear immediately. Fix it by removing overflow: hidden, switching to overflow: clip (which does not affect shadows in all browsers), or adding enough padding to the container so the shadow falls inside rather than outside it.

Make a full-width header with no horizontal scroll

Open the page and scroll down until the vertical scrollbar appears. Does a horizontal scrollbar also appear at the bottom?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Make a Full-Width Header</title>
  <style>
    body {
      margin: 0;
    }
    .header {
      width: 100vw;
      background: steelblue;
      color: white;
      padding: 16px;
      box-sizing: border-box;
    }
    .content {
      height: 200vh; /* tall enough to force a vertical scrollbar */
      padding: 16px;
    }
  </style>
</head>
<body>
  <div class="header">Full-width header</div>
  <div class="content">
    <h1>Make a full-width header</h1>
    <p>Scroll down so the vertical scrollbar appears.
       Does a horizontal scrollbar also appear at the bottom of the page?</p>
  </div>
</body>
</html>
Show explanation

The bug is width: 100vw. The vw unit measures the full viewport width including the scrollbar gutter. When a vertical scrollbar is present (about 15–17px wide on most platforms), the available content area is narrower than 100vw. The header overflows by the scrollbar width, which causes the browser to add a horizontal scrollbar.

Shows: the difference between 100vw (viewport width) and 100% (width of the containing block, which excludes the scrollbar).

To find it: resize the window so no vertical scrollbar is present—the horizontal scrollbar disappears. Add the scrollbar back—it returns. Replace width: 100vw with width: 100% on the header. Because block-level elements default to width: 100%, you can also simply remove the width declaration entirely.

Show a dropdown menu on top of a card

Open the page. The dark dropdown should appear in front of the red overlay. Does it?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Show a Dropdown Menu on Top of a Card</title>
  <style>
    .card {
      position: relative;
      background: white;
      border: 1px solid #ccc;
      padding: 16px;
      width: 200px;
      opacity: 0.99;
    }
    .dropdown {
      position: absolute;
      top: 40px;
      left: 0;
      background: #333;
      color: white;
      padding: 8px 12px;
      z-index: 200;
    }
    .overlay {
      position: relative;
      z-index: 10;
      background: tomato;
      color: white;
      padding: 8px;
      margin-top: -30px;
    }
  </style>
</head>
<body>
  <h1>Show a dropdown menu on top of a card</h1>
  <p>The dark dropdown should appear in front of the red overlay.
     Does it?</p>
  <div class="card">
    Card title
    <div class="dropdown">Dropdown item</div>
  </div>
  <div class="overlay">Overlay (z-index: 10)</div>
</body>
</html>
Show explanation

The bug is opacity: 0.99 on .card. Any opacity value less than 1 creates a new stacking context. Inside that context, the dropdown's z-index: 200 is compared only against other descendants of .card—not against elements outside it. The .card stacking context as a whole is compared to .overlay using .card's z-index, which is auto. auto loses to z-index: 10, so the entire card (including the dropdown) is painted behind the overlay.

Shows: how opacity, transform, filter, and a few other properties create new stacking contexts that trap child z-indices inside them.

To find it: remove opacity: 0.99 from .card in DevTools. The dropdown will jump in front of the overlay immediately. If you need transparency, apply it only to the card's background color using rgba() rather than the opacity property, which affects all descendants.

Set consistent font sizes with relative units

Open the page. Each nested box uses font-size: 1.5em. Should they all be the same size?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Set Consistent Font Sizes with Relative Units</title>
  <style>
    body {
      font-size: 16px;
    }
    .level-1 {
      font-size: 1.5em;
    }
    .level-2 {
      font-size: 1.5em;
    }
    .level-3 {
      font-size: 1.5em;
    }
  </style>
</head>
<body>
  <h1>Set consistent font sizes with relative units</h1>
  <p>Each box below uses <code>font-size: 1.5em</code>. Should they all be the same size?</p>
  <div class="level-1">
    Level 1 (1.5em)
    <div class="level-2">
      Level 2 (1.5em)
      <div class="level-3">
        Level 3 (1.5em)
      </div>
    </div>
  </div>
</body>
</html>
Show explanation

The bug is using em where rem is needed. An em is relative to the element's own computed font size—which is itself resolved from the parent's font size. Each level of nesting multiplies: 16px × 1.5 = 24px, then 24px × 1.5 = 36px, then 36px × 1.5 = 54px. The text grows exponentially rather than staying constant.

Shows: the difference between em (relative to the element's font size, which cascades) and rem (relative to the root element's font size, which stays fixed).

To find it: in DevTools, check the computed font-size for each nested element. Each will be 1.5 times the previous. Replace em with rem in all three rules so every element resolves against the root 16px rather than its parent.

Highlight every other row in a table

Open the page. The blue shading should appear on rows 2 and 4 of the data (Banana and Date). Which rows are actually shaded?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Highlight Every Other Row in a Table</title>
  <style>
    table {
      border-collapse: collapse;
      width: 300px;
    }
    th, td {
      border: 1px solid #ccc;
      padding: 6px 10px;
      text-align: left;
    }
    th {
      background: steelblue;
      color: white;
    }
    tr:nth-child(even) {
      background: #d0e8f7;
    }
  </style>
</head>
<body>
  <h1>Highlight every other row in a table</h1>
  <p>The shading should appear on rows 2 and 4 of the data (Banana and Date).
     Which rows are actually shaded?</p>
  <table>
    <thead>
      <tr><th>Fruit</th><th>Color</th></tr>
    </thead>
    <tbody>
      <tr><td>Apple</td><td>Red</td></tr>
      <tr><td>Banana</td><td>Yellow</td></tr>
      <tr><td>Cherry</td><td>Red</td></tr>
      <tr><td>Date</td><td>Brown</td></tr>
    </tbody>
  </table>
</body>
</html>
Show explanation

The bug is that tr:nth-child(even) counts all <tr> elements in the table, including the header row inside <thead>. The header is child 1, so the even-numbered children are header+1 (Apple, child 2), Cherry (child 4), which makes Apple and Cherry shaded instead of Banana and Date.

Shows: how :nth-child counts all siblings of the same parent, not just elements of the same type, and how <thead> and <tbody> affect the count.

To find it: add outline: 2px solid red to thead tr in DevTools and count the rows. Fix the selector by scoping it to the body rows: tbody tr:nth-child(even). Because <tbody> contains only the data rows, the count starts fresh at Apple (child 1), making Banana (child 2) and Date (child 4) the even-numbered rows.
