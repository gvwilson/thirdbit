CSS

CSS controls how HTML elements look on screen. Its rules seem simple at first—select an element, set some properties—but a handful of non-obvious behaviors trip up almost every developer at least once. Open each example directly in a browser to see the problem in action.

Change the color of a heading

Open the page. The heading should be red. Is it?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Change the Color of a Heading</title>
  <style>
    h1 {
      font-color: tomato;
    }
  </style>
</head>
<body>
  <h1>This heading should be red</h1>
  <p>Is it?</p>
</body>
</html>
Show explanation

The bug is font-color, which is not a valid CSS property. The correct property is color. When a browser encounters a property name it does not recognize, it discards the entire declaration silently—no error appears on the page or in the console unless you check the DevTools Styles panel.

Shows: that CSS ignores unknown property names without warning, and that the property for text color is color, not font-color.

To find it: open DevTools, select the <h1>, and look at the Styles panel. The font-color: tomato declaration will have a yellow warning triangle or a strikethrough indicating it is invalid. Change font-color to color.

Make a heading larger

Open the page. The heading should be noticeably larger than the paragraph below it. Is it?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Make a Heading Larger</title>
  <style>
    h1 {
      font-size: 48;
    }
  </style>
</head>
<body>
  <h1>This heading should be 48px</h1>
  <p>Is it noticeably larger than the paragraph text?</p>
</body>
</html>
Show explanation

The bug is a missing unit. font-size: 48 is invalid—CSS requires a unit such as px, em, or rem for length values (zero is the only exception). The browser discards the invalid declaration and falls back to the default heading size.

Shows: that most CSS length values require an explicit unit and that 48 and 48px are completely different things to the browser.

To find it: open DevTools, select the <h1>, and look at the Styles panel. The font-size: 48 line will be flagged as invalid. Change it to font-size: 48px.

Style an error message

Open the page. The message should have a red background and white text. Does it?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Style an Error Message</title>
  <style>
    .warning {
      color: white;
      background: tomato;
      padding: 8px 12px;
    }
  </style>
</head>
<body>
  <h1>Style an error message</h1>
  <p class="error">Something went wrong. Please try again.</p>
  <p>The message above should have a red background and white text. Does it?</p>
</body>
</html>
Show explanation

The bug is a mismatch between the selector and the HTML. The CSS targets .warning but the element has class="error". CSS class selectors are case-sensitive and must match exactly. No element in the document has the class warning, so the rule applies to nothing.

Shows: how to connect CSS rules to HTML elements via class names, and that a missing style is often a typo in either the selector or the HTML attribute.

To find it: compare the selector in the stylesheet to the class attribute in the HTML. Either change .warning to .error in the CSS, or change class="error" to class="warning" in the HTML.

Style a button with several properties

Open the page. The button should have white text and visible padding around it. Does it?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Style a Button with Several Properties</title>
  <style>
    .btn {
      background: steelblue;
      color: white
      padding: 10px 20px;
      font-size: 1rem;
      border: none;
      cursor: pointer;
    }
  </style>
</head>
<body>
  <h1>Style a button with several properties</h1>
  <p>The button should have white text and visible padding. Does it?</p>
  <button class="btn">Click me</button>
</body>
</html>
Show explanation

The bug is a missing semicolon after color: white. CSS declarations inside a rule are separated by semicolons. When the semicolon is absent, the parser treats the next line as a continuation of the same declaration. color: white padding: 10px 20px is not a valid value, so the browser discards the entire declaration—losing both the color and the padding.

Shows: how a single missing semicolon can silently drop multiple declarations and why CSS parse errors do not stop the rest of the stylesheet from loading.

To find it: open DevTools, select the button, and look at the Styles panel. Both color and padding will be absent or flagged. Add the missing semicolon after color: white.

Set the width of a label

Open the page. The blue label should be 200px wide. Is it wider than its text?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Set the Width of a Label</title>
  <style>
    .label {
      width: 200px;
      background: steelblue;
      color: white;
      padding: 4px;
    }
  </style>
</head>
<body>
  <h1>Set the width of a label</h1>
  <p>The blue label should be 200px wide. Is it wider than the text inside it?</p>
  <p><span class="label">Status: OK</span></p>
</body>
</html>
Show explanation

The bug is that <span> is an inline element. Inline elements flow with text and ignore width and height declarations. The label is exactly as wide as its content regardless of what the CSS says.

Shows: the inline/block distinction and that width only affects block-level or replaced elements (such as <img>).

To find it: in DevTools, select the <span> and look at the box model diagram. The width will equal the text width, not 200px. Add display: inline-block or display: block to the .label rule to make width take effect.

Add a background image to a panel

Open the page. The panel should have a blue background (shown when the image is missing). Is it blue?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Add a Background Image to a Panel</title>
  <style>
    .panel {
      background-color: steelblue;
      background: url("nonexistent.png") no-repeat center center;
      color: white;
      padding: 24px;
      width: 300px;
    }
  </style>
</head>
<body>
  <h1>Add a background image to a panel</h1>
  <p>The panel should have a blue background (visible when the image is missing).
     Is it blue?</p>
  <div class="panel">Panel content</div>
</body>
</html>
Show explanation

The bug is declaration order. The background shorthand property sets every background sub-property at once, including background-color, resetting it to transparent. Because background appears after background-color in the rule, it overwrites the blue color. The image URL does not exist, and the fallback color is now transparent rather than blue.

Shows: that background is a shorthand that resets all background sub-properties, and that rule order within a block matters.

To find it: in DevTools, look at the computed background-color for the panel. It will be transparent. Either move background-color after the background line, or include the color directly in the shorthand: background: url("...") no-repeat center center steelblue.

Hide a notice without moving other content

Open the page. Compare where the paragraph sits relative to where it would be if the notice were visible. Did the paragraph move up?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Hide a Notice Without Moving Other Content</title>
  <style>
    .notice {
      background: gold;
      padding: 10px;
      margin-bottom: 12px;
      display: none;
    }
  </style>
</head>
<body>
  <h1>Hide a notice without moving other content</h1>
  <div class="notice">Important notice: please read before continuing.</div>
  <p>This paragraph should stay in the same position whether the notice is visible or not.
     Does it move when the notice is hidden?</p>
</body>
</html>
Show explanation

The bug is using display: none when visibility: hidden is needed. display: none removes the element from the layout entirely—other elements flow in as if it were not there. visibility: hidden hides the element visually but reserves its space, so surrounding elements do not shift.

Shows: the difference between display: none (removes from flow) and visibility: hidden (invisible but still occupies space).

To find it: in DevTools, toggle display: none off. The notice reappears and the paragraph drops back to its original position, confirming the layout was collapsing. Change display: none to visibility: hidden to hide the notice while preserving the layout.

Center a button on the page

Open the page. The button should be centered horizontally on the page. Is it?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Center a Button on the Page</title>
  <style>
    .btn {
      display: block;
      padding: 10px 24px;
      background: steelblue;
      color: white;
      border: none;
      font-size: 1rem;
      cursor: pointer;
      text-align: center;
    }
  </style>
</head>
<body>
  <h1>Center a button on the page</h1>
  <p>The button below should be centered horizontally. Is it?</p>
  <button class="btn">Submit</button>
</body>
</html>
Show explanation

The bug is applying text-align: center to the button itself. text-align controls the alignment of inline content—text and inline elements—inside the element it is applied to. On the button, it centers the label text within the button's box. It does nothing to position the button on the page.

Shows: that text-align affects content inside an element, not the element's position in its parent, and that centering a block-level element requires a different technique.

To find it: look at the button in DevTools. The text inside is centered, but the button stretches edge to edge. Move text-align: center to the parent element (e.g., body or a wrapper <div>). A block-level child inherits the alignment hint and centers within its parent.

Center a card horizontally

Open the page. The blue card should be centered with equal space on each side. Is it?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Center a Card Horizontally</title>
  <style>
    .card {
      margin: 0 auto;
      background: steelblue;
      color: white;
      padding: 20px;
    }
  </style>
</head>
<body>
  <h1>Center a card horizontally</h1>
  <p>The blue card should be centered with space on both sides. Is it?</p>
  <div class="card">Centered card</div>
</body>
</html>
Show explanation

The bug is a missing width. margin: 0 auto centers a block element by dividing leftover horizontal space equally between the left and right margins. But a block element without an explicit width expands to fill its container entirely, leaving no leftover space to distribute. The card is full-width and auto resolves to zero on both sides.

Shows: that margin: 0 auto only works when the element is narrower than its container, which requires an explicit width.

To find it: in DevTools, look at the computed margin-left and margin-right—both will be 0. Add width: 300px (or any value less than the container width) to .card so the browser has remaining space to divide equally.

Color a link differently on hover

Click the link to visit it, then come back and hover over it. Does it turn red?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Color a Link Differently on Hover</title>
  <style>
    a:hover {
      color: tomato;
    }
    a:visited {
      color: purple;
    }
    a:link {
      color: steelblue;
    }
  </style>
</head>
<body>
  <h1>Color a link differently on hover</h1>
  <p>Click the link below to visit it, then come back.
     Hover over it. Does it turn red on hover, or does it stay purple?</p>
  <p><a href="https://example.com" target="_blank">Visit example.com</a></p>
</body>
</html>
Show explanation

The bug is rule order. All three selectors (a:hover, a:visited, a:link) have the same specificity, so the browser uses source order as a tiebreaker—the last matching rule wins. For a visited link, both a:hover and a:visited match when the cursor is over it. Because a:visited appears after a:hover in the stylesheet, it always wins and the hover color never shows. The correct order is :link, :visited, :hover, :active—often remembered as LVHA.

Shows: how the CSS cascade uses source order when specificity is equal, and why link pseudo-classes must be declared in LVHA order.

To find it: reorder the rules so a:link comes first, then a:visited, then a:hover. With that order, :hover appears later and overrides :visited when the cursor is present.

Fit a panel inside a container

Open the page. The blue panel should sit entirely within the grey container. Does it, or does it overflow to the right?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Fit a Panel Inside a Container</title>
  <style>
    .container {
      width: 300px;
      background: #ddd;
      padding: 0;
    }
    .panel {
      width: 100%;
      padding: 20px;
      background: steelblue;
      color: white;
    }
  </style>
</head>
<body>
  <h1>Fit a panel inside a container</h1>
  <p>The blue panel should sit entirely within the grey container.
     Does it fit, or does it stick out to the right?</p>
  <div class="container">
    <div class="panel">Panel content</div>
  </div>
</body>
</html>
Show explanation

The bug is box-sizing: content-box, which is the default. When width: 100% is set, the browser assigns 100% of the container's width to the content area alone. Padding is then added on top, making the element wider than the container—300px content + 20px left padding + 20px right padding = 340px total.

Shows: the CSS box model and the difference between content-box and border-box.

To find it: open DevTools, select the panel, and look at the box model diagram in the Computed tab. The total rendered width will exceed the container's width. Add box-sizing: border-box to the .panel rule so that padding is included inside the declared width rather than added to it.

Override a color in a stylesheet

Open the page. Both elements use the .highlight class. Are they both red?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Override a Color in a Stylesheet</title>
  <style>
    nav.highlight {
      color: gray;
    }

    .highlight {
      color: red;
    }
  </style>
</head>
<body>
  <h1>Override a color in a stylesheet</h1>
  <p>The navigation bar below should be red. Is it?</p>
  <nav class="highlight">Site Navigation</nav>
  <p class="highlight">This paragraph uses the same class and should also be red.</p>
</body>
</html>
Show explanation

The bug is a specificity conflict. The selector nav.highlight targets an element type plus a class, giving it specificity (0,1,1). The later rule .highlight is class-only, specificity (0,1,0). CSS resolves conflicts by specificity first and source order second, so the less-specific later rule loses to the more-specific earlier rule even though it appears afterward in the file.

Shows: how the browser calculates specificity and why rule order alone does not determine which style wins.

To find it: in DevTools, click the <nav> element and examine the Styles panel. The .highlight { color: red } rule will be shown with a strikethrough, indicating it was overridden. To fix it, either change .highlight to nav.highlight, .highlight or restructure the stylesheet so both elements share a single rule without the extra type selector.

Add space between sections

Open the page. Section Two has margin-top: 60px and Section One has margin-bottom: 20px. Measure the gap. Is it 80px?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Add Space Between Sections</title>
  <style>
    .section {
      background: #b3d9f7;
      padding: 10px;
      margin-bottom: 20px;
    }
    .section.large-gap {
      margin-top: 60px;
    }
  </style>
</head>
<body>
  <h1>Add space between sections</h1>
  <p>The second section has <code>margin-top: 60px</code> and the first has
     <code>margin-bottom: 20px</code>. How much space actually appears between them?</p>
  <div class="section">Section One</div>
  <div class="section large-gap">Section Two (margin-top: 60px)</div>
  <div class="section">Section Three</div>
</body>
</html>
Show explanation

The bug is margin collapsing. When two block elements are adjacent siblings, their vertical margins do not add up—they collapse into a single margin equal to the larger of the two values. The gap between Section One and Section Two is 60px (the larger margin), not 80px (the sum).

Shows: vertical margin collapsing between adjacent siblings and why the gap is never the sum of both margins.

To find it: in DevTools, hover over the gap between sections. The box model overlay will show only the dominant margin. To get the full 80px gap, convert one of the margins to padding on the element itself, or add display: flow-root to a wrapper to create a new block formatting context that prevents collapsing.

Put a border around an image and caption

Open the page. The blue border should enclose the image placeholder and the caption. Does the border appear to wrap them?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Put a Border Around an Image and Caption</title>
  <style>
    .card {
      border: 3px solid steelblue;
      padding: 8px;
      width: 260px;
    }
    img {
      float: left;
      width: 80px;
      height: 80px;
      background: #aaa;
    }
    .caption {
      float: left;
      margin-left: 12px;
      line-height: 80px;
    }
  </style>
</head>
<body>
  <h1>Put a border around an image and caption</h1>
  <p>The blue border should wrap around both the image placeholder and the caption.
     Does it?</p>
  <div class="card">
    <img alt="placeholder">
    <span class="caption">Photo caption here</span>
  </div>
</body>
</html>
Show explanation

The bug is that floated elements are removed from the normal flow. The parent .card has no in-flow children to give it height, so it collapses to zero height (plus padding). The blue border wraps only the padding, not the floated image or caption.

Shows: how floats interact with parent element height and the classic clearfix problem.

To find it: in DevTools, select .card and observe its height is 16px (just padding). Fix it by adding overflow: auto to .card, which forces the parent to contain its floated children. Alternatively, add a clearing element after the floats, or switch to flexbox or grid.

Show a tooltip above an icon

Open the page. The dark tooltip should appear in front of the red banner. Does it?

i

<html lang="en">
<head>
  <meta charset="utf-8">

  <title>Show a Tooltip Above an Icon</title>
  <style>
    .icon {
      display: inline-block;
      width: 40px;
      height: 40px;
      background: steelblue;
      color: white;
      text-align: center;
      line-height: 40px;
      font-size: 20px;
    }
    .tooltip {
      background: #333;
      color: white;
      padding: 4px 8px;
      z-index: 100;
    }
    .banner {
      position: relative;
      z-index: 10;
      background: tomato;
      color: white;
      padding: 8px;
      margin-top: -10px;
    }
  </style>
</head>
<body>
  <h1>Show a tooltip above an icon</h1>
  <p>The dark tooltip should appear in front of the red banner. Does it?</p>

Show explanation

The .tooltip element has no position property, so it remains position: static and the z-index: 100 declaration is ignored entirely. Shows: the requirement that z-index needs a non-static position to take effect.

Add a decorative bullet to each list item

Open the page. Each list item should have a gold star to its left. Do any stars appear?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Add a Decorative Bullet to Each List Item</title>
  <style>
    ul {
      list-style: none;
      padding-left: 0;
    }
    li::before {
      color: gold;
      font-size: 1.2em;
      margin-right: 6px;
    }
  </style>
</head>
<body>

Show explanation

Fill half a page with a colored banner

Open the page. The blue banner should fill half the grey wrapper. Is the wrapper visible at all?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Fill Half a Page with a Colored Banner</title>
  <style>
    .wrapper {
      background: #eee;
    }
    .banner {
      height: 50%;
      background: steelblue;
    }
  </style>
</head>
<body>
  <h1>Fill half a page with a colored banner</h1>
  <p>The blue banner should fill half the grey wrapper. Is the wrapper visible at all?</p>
  <div class="wrapper">
    <div class="banner"></div>
  </div>
  <p>Text below the wrapper.</p>
</body>
</html>
Show explanation

The bug is that percentage heights are resolved relative to the parent's computed height. The .wrapper has no explicit height—its height is auto, which the browser calculates from its in-flow content. The browser cannot resolve 50% of auto, so it treats the child's height as zero. The blue banner has no height and is invisible; the grey wrapper collapses to nothing as well.

Shows: how percentage heights require an explicit height on the ancestor and why auto breaks the chain.

To find it: in DevTools, select .banner and look at its computed height: 0px. To fix it, give .wrapper an explicit height (e.g., height: 300px), use viewport units (height: 50vh) on the banner directly, or switch to a flexbox or grid layout.

Animate a button color on hover

Open the page and hover over the button. Does the background color fade smoothly to red, or does it snap?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Animate a Button Color on Hover</title>
  <style>
    .btn {
      padding: 10px 20px;
      background: steelblue;
      color: white;
      border: none;
      cursor: pointer;
      font-size: 1rem;
      transition: color 0.4s ease;
    }
    .btn:hover {
      background-color: tomato;
    }
  </style>
</head>
<body>
  <h1>Animate a button color on hover</h1>
  <p>Hover over the button. Does the background color fade smoothly to red,
     or does it snap?</p>
  <button class="btn">Hover me</button>
</body>
</html>
Show explanation

The bug is that transition: color 0.4s ease animates the color property (text color), not background-color. The hover rule changes background-color, which has no transition declared, so it jumps instantly. Only a text-color change would animate here, but the hover rule does not change text color, so nothing animates at all.

Shows: how transitions must name the exact CSS property being changed and that color and background-color are distinct.

To find it: in DevTools, look at the .btn rule and the hover rule side by side. The transitioned property (color) does not match the property changed on hover (background-color). Fix it by changing the transition to background-color 0.4s ease or using transition: all 0.4s ease to cover every animated property.

      min-width: 140px;

      background: steelblue;
      color: white;
      margin: 8px;
      display: flex;
      align-items: center;
      justify-content: center;
    }
  </style>
</head>
<body>
  <h1>Wrap a grid of cards to a new row</h1>
  <p>Five cards should wrap onto two rows inside the dashed box.
     Do they wrap, or do they overflow to the right?</p>
  <div class="grid">
    <div class="card">Card 1</div>
    <div class="card">Card 2</div>
    <div class="card">Card 3</div>
    <div class="card">Card 4</div>
    <div class="card">Card 5</div>
  </div>
</body>
</html>
Show explanation

The bug is that flex-wrap defaults to nowrap. All flex children are forced onto a single line regardless of the container's width. When the total minimum size of the children exceeds the container, they shrink as much as they can and then overflow.

Shows: the default flex-wrap: nowrap behavior and why adding flex-wrap: wrap is usually necessary for responsive grid-style layouts.

To find it: open DevTools and select the .grid container. In the Layout panel, the flex wrapping option will show nowrap. Add flex-wrap: wrap to .grid so children that do not fit on one line flow onto subsequent lines.

Add a shadow to a card

Open the page. The white card should have a visible drop shadow. Can you see any shadow?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Add a Shadow to a Card</title>
  <style>
    .container {
      width: 300px;
      background: #f0f0f0;
      padding: 20px;
      overflow: hidden;
    }
    .card {
      background: white;
      padding: 16px;
      box-shadow: 4px 4px 12px rgba(0, 0, 0, 0.4);
    }
  </style>
</head>
<body>
  <h1>Add a shadow to a card</h1>
  <p>The white card should have a visible drop shadow. Can you see any shadow?</p>
  <div class="container">
    <div class="card">Card content goes here.</div>
  </div>
</body>
</html>
Show explanation

The bug is overflow: hidden on the container. box-shadow is painted outside the element's border-box, in the space between the card and the container's edge. Because overflow: hidden clips everything that extends beyond the container's boundary, the shadow—which lives in that clipped region—is invisible.

Shows: that overflow: hidden clips more than just scrollable content; it also clips effects such as box-shadow and outline that paint outside the element's box.

To find it: temporarily set the container to overflow: visible in DevTools. The shadow will reappear immediately. Fix it by removing overflow: hidden, switching to overflow: clip (which does not affect shadows in all browsers), or adding enough padding to the container so the shadow falls inside rather than outside it.

Make a full-width header with no horizontal scroll

Open the page and scroll down until the vertical scrollbar appears. Does a horizontal scrollbar also appear at the bottom?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Make a Full-Width Header</title>
  <style>
    body {
      margin: 0;
    }
    .header {
      width: 100vw;
      background: steelblue;
      color: white;
      padding: 16px;
      box-sizing: border-box;
    }
    .content {
      height: 200vh; /* tall enough to force a vertical scrollbar */
      padding: 16px;
    }
  </style>
</head>
<body>
  <div class="header">Full-width header</div>
  <div class="content">
    <h1>Make a full-width header</h1>
    <p>Scroll down so the vertical scrollbar appears.
       Does a horizontal scrollbar also appear at the bottom of the page?</p>
  </div>
</body>
</html>
Show explanation

The bug is width: 100vw. The vw unit measures the full viewport width including the scrollbar gutter. When a vertical scrollbar is present (about 15–17px wide on most platforms), the available content area is narrower than 100vw. The header overflows by the scrollbar width, which causes the browser to add a horizontal scrollbar.

Shows: the difference between 100vw (viewport width) and 100% (width of the containing block, which excludes the scrollbar).

To find it: resize the window so no vertical scrollbar is present—the horizontal scrollbar disappears. Add the scrollbar back—it returns. Replace width: 100vw with width: 100% on the header. Because block-level elements default to width: 100%, you can also simply remove the width declaration entirely.

Show a dropdown menu on top of a card

Open the page. The dark dropdown should appear in front of the red overlay. Does it?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Show a Dropdown Menu on Top of a Card</title>
  <style>
    .card {
      position: relative;
      background: white;
      border: 1px solid #ccc;
      padding: 16px;
      width: 200px;
      opacity: 0.99;
    }
    .dropdown {
      position: absolute;
      top: 40px;
      left: 0;
      background: #333;
      color: white;
      padding: 8px 12px;
      z-index: 200;
    }
    .overlay {
      position: relative;
      z-index: 10;
      background: tomato;
      color: white;
      padding: 8px;
      margin-top: -30px;
    }
  </style>
</head>
<body>
  <h1>Show a dropdown menu on top of a card</h1>
  <p>The dark dropdown should appear in front of the red overlay.
     Does it?</p>
  <div class="card">
    Card title
    <div class="dropdown">Dropdown item</div>
  </div>
  <div class="overlay">Overlay (z-index: 10)</div>
</body>
</html>
Show explanation

The bug is opacity: 0.99 on .card. Any opacity value less than 1 creates a new stacking context. Inside that context, the dropdown's z-index: 200 is compared only against other descendants of .card—not against elements outside it. The .card stacking context as a whole is compared to .overlay using .card's z-index, which is auto. auto loses to z-index: 10, so the entire card (including the dropdown) is painted behind the overlay.

Shows: how opacity, transform, filter, and a few other properties create new stacking contexts that trap child z-indices inside them.

To find it: remove opacity: 0.99 from .card in DevTools. The dropdown will jump in front of the overlay immediately. If you need transparency, apply it only to the card's background color using rgba() rather than the opacity property, which affects all descendants.

Set consistent font sizes with relative units

Open the page. Each nested box uses font-size: 1.5em. Should they all be the same size?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Set Consistent Font Sizes with Relative Units</title>
  <style>
    body {
      font-size: 16px;
    }
    .level-1 {
      font-size: 1.5em;
    }
    .level-2 {
      font-size: 1.5em;
    }
    .level-3 {
      font-size: 1.5em;
    }
  </style>
</head>
<body>
  <h1>Set consistent font sizes with relative units</h1>
  <p>Each box below uses <code>font-size: 1.5em</code>. Should they all be the same size?</p>
  <div class="level-1">
    Level 1 (1.5em)
    <div class="level-2">
      Level 2 (1.5em)
      <div class="level-3">
        Level 3 (1.5em)
      </div>
    </div>
  </div>
</body>
</html>
Show explanation

The bug is using em where rem is needed. An em is relative to the element's own computed font size—which is itself resolved from the parent's font size. Each level of nesting multiplies: 16px × 1.5 = 24px, then 24px × 1.5 = 36px, then 36px × 1.5 = 54px. The text grows exponentially rather than staying constant.

Shows: the difference between em (relative to the element's font size, which cascades) and rem (relative to the root element's font size, which stays fixed).

To find it: in DevTools, check the computed font-size for each nested element. Each will be 1.5 times the previous. Replace em with rem in all three rules so every element resolves against the root 16px rather than its parent.

Highlight every other row in a table

Open the page. The blue shading should appear on rows 2 and 4 of the data (Banana and Date). Which rows are actually shaded?

i
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Highlight Every Other Row in a Table</title>
  <style>
    table {
      border-collapse: collapse;
      width: 300px;
    }
    th, td {
      border: 1px solid #ccc;
      padding: 6px 10px;
      text-align: left;
    }
    th {
      background: steelblue;
      color: white;
    }
    tr:nth-child(even) {
      background: #d0e8f7;
    }
  </style>
</head>
<body>
  <h1>Highlight every other row in a table</h1>
  <p>The shading should appear on rows 2 and 4 of the data (Banana and Date).
     Which rows are actually shaded?</p>
  <table>
    <thead>
      <tr><th>Fruit</th><th>Color</th></tr>
    </thead>
    <tbody>
      <tr><td>Apple</td><td>Red</td></tr>
      <tr><td>Banana</td><td>Yellow</td></tr>
      <tr><td>Cherry</td><td>Red</td></tr>
      <tr><td>Date</td><td>Brown</td></tr>
    </tbody>
  </table>
</body>
</html>
Show explanation

The bug is that tr:nth-child(even) counts all <tr> elements in the table, including the header row inside <thead>. The header is child 1, so the even-numbered children are header+1 (Apple, child 2), Cherry (child 4), which makes Apple and Cherry shaded instead of Banana and Date.

Shows: how :nth-child counts all siblings of the same parent, not just elements of the same type, and how <thead> and <tbody> affect the count.

To find it: add outline: 2px solid red to thead tr in DevTools and count the rows. Fix the selector by scoping it to the body rows: tbody tr:nth-child(even). Because <tbody> contains only the data rows, the count starts fresh at Apple (child 1), making Banana (child 2) and Date (child 4) the even-numbered rows.
