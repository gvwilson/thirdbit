Conclusion

looking back and next steps

Tactics

Every exercise in this book uses one or more of the following tactics to locate a bug. Each tactic is a concrete action you can take in seconds; none of them requires guessing.

  1. Run the code with the smallest input you can verify entirely by hand, then check the output against your manual count or calculation.
  2. Print type(value) when a comparison unexpectedly fails — a mismatch between str and int, or String and Date, explains why == returns False.
  3. Print repr(string) to reveal invisible characters such as trailing spaces or embedded newlines that look correct in plain print output.
  4. Call a function twice with identical arguments and compare both return values — if they differ, the function carries state between calls.
  5. Read a traceback from bottom to top: the last frame names the line that raised the error; the frames above it show what triggered that line.
  6. Introduce a deliberate bug in the code under test, then rerun the test suite — if the test still passes, it contains no assertion that can detect wrong output.
  7. Run each test in isolation with pytest path::test_name to verify it does not depend on another test having run first.
  8. Replace a broad except: pass with logging.exception("caught") to see the actual exception type and message being swallowed.
  9. Check the exact stored values in a column before writing a filter — use SELECT DISTINCT col or .unique() to confirm the literal strings or numbers the data actually contains.
  10. Count rows before and after each transformation step to detect rows that were silently dropped or duplicated.
  11. Run python -m cProfile -s cumulative script.py | head -20 to find the slow line instead of guessing which part of the code needs optimization.
  12. Add print("mock was called") as the first line of a mock to confirm it is actually intercepting the real call.
  13. Open the browser's Network tab and Console tab to see what requests were sent, what status codes came back, and what errors were logged.
  14. Echo or print the exact arguments a shell command receives before running it on real data — use echo $var or replace a command with echo command to preview what the shell will execute.
  15. Check HTTP status codes before parsing the response body — a non-200 response often returns HTML, not JSON, so the parse error is a symptom, not the cause.
  16. Insert .collect() after each step in a Polars lazy pipeline to pin down which transformation is raising an error, rather than waiting for the final collect.
  17. Test a regular expression against known-invalid inputs, not just valid ones, to verify the pattern rejects what it should.
  18. Sketch the directory tree on paper and count directory levels before running a command that uses a relative path such as ../...
